# Gist — мутабельная модель понимания задачи

## Проблема

Сессионные саммари — это event log (что произошло). Агенту не хватает **state** — целостного понимания задачи, которое человек держит в голове между сессиями. Факты атомарны и не дают картины. Compact описывает одну сессию, не arc задачи.

Когнитивная основа:
- **Gist vs Verbatim** (Fuzzy Trace Theory) — люди предпочитают gist (минимально точное представление для принятия решений), а не verbatim (точные детали)
- **Situation Awareness** (Endsley) — три уровня: perception → comprehension → projection. Саммари дают только perception, нужны comprehension + projection
- **Schema theory** — схемы перезаписываются при каждом новом опыте, а не накапливаются

## Структура

```markdown
---
topic: VDPresent transitions
project: vdpresent
created: 2026-04-01
updated: 2026-04-08
commit: abc1234
tags: [transitions, animation, interactive]
---

## Vision
Каким должен быть результат. Критерий "готово".
Стабильный слой — меняется редко.

## Model
Как система должна работать на языке требований, не кода.
Поведение, контракты, инварианты.
Обновляется при смене архитектуры.
```

### Stored layers (в файле)

| Слой | Что хранит | Волатильность | Кто обновляет |
|------|-----------|---------------|---------------|
| **Vision** | Цель, критерий готовности, как должно ощущаться | Редко | Только explicit (пользователь или агент с подтверждением) |
| **Model** | Спецификация поведения на человеческом языке — контракты, инварианты, бизнес-правила | При смене архитектуры | Агент в диалоге когда пользователь обсуждает архитектуру. LLM-auto запрещено |

### Synthesized layers (при /load, не хранятся)

| Слой | Что хранит | Источник |
|------|-----------|---------|
| **Meaning** | Где мы сейчас относительно Vision + Model, что работает, что нет и почему | LLM синтез из: gist + `git log {commit}..HEAD` + recent sessions |
| **Direction** | Текущая гипотеза, следующий шаг | LLM синтез из того же |

### Frontmatter

- `topic` — название задачи/направления (не обязательно = проект)
- `project` — проект (для scoping)
- `created` / `updated` — даты создания и последнего обновления
- `commit` — хэш коммита на момент последнего обновления (для `git log {commit}..HEAD` при /load)
- `tags` — для поиска и автоматического matching

## Ключевые решения

1. **Topic, не project** — в проекте может быть несколько активных задач, каждая со своим gist
2. **Нет auto-rewrite** — Vision и Model не перезаписываются LLM автоматически, нет drift
3. **Нет Log секции** — мотивации решений хранятся в Facts, git history покрывает хронологию коммитов
4. **commit в frontmatter** — позволяет обнаружить "работал без агента" и показать diff
5. **Meaning/Direction не хранятся** — генерируются свежими при /load, всегда актуальны
